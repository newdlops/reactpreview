# 기여 지침

## 변경 전 판단

새 코드를 추가하기 전에 어느 책임이 변하는지 먼저 정합니다.

- React 프리뷰의 의미와 데이터라면 `domain`
- 컴파일과 저장의 작업 순서라면 `application`
- esbuild 또는 VS Code API 세부사항이라면 해당 `adapter`
- 패널, 이벤트, HTML 상태라면 `presentation`
- 객체 생성과 명령 등록이라면 `extension.ts`

한 파일이 두 개 이상의 책임을 설명해야 한다면 분리합니다. 단순히 파일 수를 줄이기 위해 계층을
합치지 않습니다.

## 코드 기준

1. 모든 텍스트 파일은 물리적으로 1,000줄 이하이며 생성 파일도 예외로 두지 않습니다. npm
   lockfile은 `.npmrc`에서 생성을 막고 상위 의존성 버전을 `package.json`에 정확히 고정합니다.
2. 실제 목표는 소스 파일 200~400줄입니다. 한 파일이 반복해서 여러 이유로 바뀌면 더 일찍
   분리합니다.
3. 파일 상단 주석은 파일 책임, 계층 경계, 중요한 불변식을 설명합니다.
4. 함수와 클래스 JSDoc은 “무엇을 하는가”뿐 아니라 입력, 반환값, 던지는 오류, 중요한 부작용을
   설명합니다. 이름을 그대로 반복하는 주석은 추가하지 않습니다.
5. 공개 type과 interface의 필드는 의미와 단위를 주석으로 남깁니다.
6. 읽기 어려운 축약, 숨은 전역 상태, boolean 의미를 알 수 없는 호출을 피합니다.
7. 외부 구현을 직접 생성하지 말고 composition root에서 port에 주입합니다.
8. 미래 가능성만을 위한 base class나 범용 utility는 만들지 않습니다. 확인된 변동점에 작은
   interface와 조합을 사용합니다.
9. 워크스페이스 입력은 신뢰할 수 없는 값으로 보고 HTML, URI, 메시지 경계에서 검증합니다.
10. 프로젝트 설정, package script, shell, 임의 VS Code command를 프리뷰 편의를 위해 실행하지
    않습니다.

JSON manifest나 lockfile처럼 주석을 허용하지 않는 형식은 인접 README 또는 아키텍처 문서에서
책임과 선택 이유를 설명합니다.

## 변경 절차

```bash
npm install
npm run watch
```

작업 중에는 영향받은 계층과 같은 경로 아래 테스트를 추가합니다. 완료 전에는 다음 한 명령을
실행합니다.

```bash
npm run check
```

이 명령은 줄 수, strict typecheck, ESLint 계층 규칙과 JSDoc, Prettier, Vitest, 실제 extension
bundle을 모두 검사합니다.

## 완료 조건

- 요청한 동작과 실패 경로가 테스트됩니다.
- 신뢰되지 않은 워크스페이스와 CSP 경계가 약해지지 않습니다.
- 사용자 프로젝트에 파일을 쓰거나 서버를 시작하지 않습니다.
- 새 의존성은 필요성과 런타임/개발 범위가 명확합니다.
- 문서의 지원 범위와 제한이 실제 구현과 일치합니다.
- 모든 유지보수 파일이 1,000줄 이하입니다.
- `npm run check`가 오류와 경고 없이 통과합니다.
